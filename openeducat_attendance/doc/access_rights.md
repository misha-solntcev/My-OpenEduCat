# Настройки прав доступа для разных групп пользователей

## Общие принципы
Система прав доступа к данным в модели `op.subject.grades` построена на групповой модели пользователей. Разграничение доступа реализовано на уровне приложения через переопределенные методы `_search` и `read`.

## Группы пользователей

### Студенты (__custom__.group_op_students)
- Могут просматривать только свои оценки
- Не имеют доступа к оценкам других студентов
- Фильтрация применяется автоматически при любом доступе к данным

### Преподаватели (__custom__.group_op_faculty)
- Могут просматривать оценки всех студентов
- Имеют доступ ко всем записям в модели `op.subject.grades`
- Не подвержены фильтрации по студентам

### Администраторы и менеджеры
- Могут просматривать оценки всех студентов
- Имеют полный доступ ко всем записям
- Не подвержены фильтрации по студентам

## Права на действия

### Просмотр данных
- Все группы пользователей могут просматривать данные в соответствии с фильтрацией
- Права на чтение реализованы через переопределенный метод `read`

### Создание записей
- Создание записей в модели `op.subject.grades` отключено в Kanban-представлении (`create="false"`)
- Записи создаются только через wizard обновления данных

### Редактирование записей
- Редактирование записей через интерфейс отключено
- Записи обновляются только через wizard обновления данных

### Удаление записей
- Удаление записей через интерфейс отключено
- Записи удаляются и создаются заново при обновлении через wizard

## Wizard обновления данных
Доступ к wizard обновления данных (`op.update.subject.grades`) ограничен группой `openeducat_attendance.group_op_attendance_manager`. Это означает, что только менеджеры посещаемости могут запускать процесс обновления агрегированных данных.

## Техническая реализация

### Определение групп
Группы пользователей определяются через внешние идентификаторы:
- `__custom__.group_op_students` - группа студентов
- `__custom__.group_op_faculty` - группа преподавателей
- `openeducat_attendance.group_op_attendance_manager` - группа менеджеров посещаемости

### Проверка принадлежности
Проверка принадлежности пользователя к группе осуществляется через свойство `groups_id` объекта `self.env.user`.

### Обработка ошибок
Реализована обработка возможных ошибок при получении ссылок на группы через метод `env.ref` с параметром `raise_if_not_found=False`.

## Особенности реализации
1. Используется комбинация фильтрации на уровне базы данных и проверки на уровне приложения
2. Реализована поддержка пользователей, принадлежащих к нескольким группам одновременно
3. Применяется логирование для отладки и мониторинга работы системы прав доступа
4. Учитываются возможные ошибки при получении групп и записей студентов
